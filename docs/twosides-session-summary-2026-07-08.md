# TwoSides 構想セッション総まとめ

> 2026-07-08。FactBase → TwoSides への方向転換、Radar、スコープ、収益モデルまで。

---

## 1. 一行コンセプト（最新）

**「バズってる争点について、多ソースで中立に土台を作り、両側の議論が戦える場。Grok に無い“投票と層の動き”が価値。」**

---

## 2. FactBase → TwoSides で何が変わったか

| | FactBase | TwoSides |
|---|---|---|
| 記事の役割 | できるだけ正しい記事 | **議論の火種・共通の土台** |
| 素材 | 一次情報必須に近い | **多ソース統合**（国内報道・声明・Wiki・公式あれば加点） |
| スコープ | 政治・法案中心 | **バズってる争点なら幅広く**（声明対立型エンタメ含む） |
| 主役 | 記事の正確さ | **スプリットスレッド + 越境評価ランキング** |
| 有料の芯 | 一次情報 FC（✅ oracle） | **議論インテリジェンス（投票・層の動き・両陣営分析）** |

## 3. 記事・Radar（供給側）

### 理想フロー

```
バズ検知 → 精査 → 多ソース収集 → 中立要約 → 出典+ラベル → 公開 → スプリット
```

### 現状コード（7割実装済み）

- **バズ検知**: `discover.ts`（Trends / Yahoo / ニュースランキング / YouTube クロス照合）
- **精査・公開**: `promote.ts`（buzzScore + 証拠十分性、ピーク3回/日）
- **素材収集**: `researchTopic()` + `fetchReportExcerpts()`（本文抜粋は取れている）
- **Tavily**: 発見のみ。本文は `fetchPageText` 経由
- **YouTube**: タイトルでバズ検知のみ。**動画/Gemini 要約は不要**

### パイプライン方針（合意）

- **メイン**: `discover → promote` のみ（バズ話題）
- **例外**: `detect → summarize` は LIVE 速報・官公庁のみ

### YouTube → 各局記事マッチ（未実装・合意）

YouTube は検知トリガー。本文はテレ朝 / TBS DIG / FNN / 日テレ / NHK 等の **同内容 Web 記事** から `fetchReportExcerpts`。

### 記事テンプレ（TwoSides 火種モード・案）

1. 何が起きた？（共通事実）
2. A 側の声明・主張 / B 側の声明・主張
3. 報道の整理（TBS/FNN 等・帰属付き）
4. どこが食い違う？（**差がある場合のみ**。裁判系は各社同型表現でも OK）
5. 論点（賛成側 / 反対側の典型論点 — 厚め）
6. 出典（種別ラベル）

タイトル **2 層**: `shareTitle`（シェア用）+ `voteQuestion`（投票用）

### 議論適性（記事にするか）

- **必須**: 国内複数媒体（`distinctNewsOutlets >= 2`）、立場を取れる争点（`debatable`）
- **不要**: 海外必須、報道食い違い必須、Wiki/公式必須
- AI は補助。主判定はルール + 素材の有無

```
優先度 = buzzScore × (debatable ? 1.0 : 0.4) × min(1, distinctNewsOutlets / 2)
```

## 4. スコープ拡大（政治以外）

### 拾う

- **声明対立型**（例: 佐藤二朗 × フジ — 組織 vs 個人、両者声明 + 各社報道）
- 賛否が自然に割れる社会話題
- スポーツも **ルール・処遇・組織判断** が争点なら可

### 拾わない

- 熱愛・結婚ゴシップ（声明対立なし）
- 試合結果だけ
- 個人私生活晒し・犯罪断定

### 現状コードとのギャップ

`filterRelevantTopics` / `OUT_OF_SCOPE_FLAGS` / `buzz-prefilter` が **芸能・ゴシップを却下** → 声明対立型を通すよう緩和が必要。

### X 投票との関係

X の「❤️=A / 🔖=B」（220万表示級）は需要の証明。TwoSides は **中身付き（声明+報道+相違点）+ スプリット + 越境評価** の上位互換。

## 5. FC と有料モデル（重要な転換）

### FactBase 時代の FC

- 一次情報 DB（法令・会議録）に照合 → TRUE/FALSE
- 他にない差別化

### TwoSides 2.0 では FC を有料の柱に **しない**

理由:

- ゴシップ・エンタメ争点では一次情報 DB が効かない
- 「AI に聞く」有料 FC は **Grok / ChatGPT と比較され負ける**
- ユーザーが欲しいのは oracle より **「どっち支持？ 議論したい」**

### FC の扱い

- **主役から降格**。削除はしない
- 政治争点: 従来 RAG（法令 DB）をボーナスとして残す
- 一般争点: 争点記事の素材との照合程度（✅ = 「出典付きで書いた」）
- oracle として売らない

## 6. 有料価値 = 議論インテリジェンス（Grok に無い）

**Grok / X が持てないデータ:**

| データ | TwoSides |
|---|---|
| 賛成 / 反対 / 中立の投票 | ✅ |
| 読前 → 読後の立場変化 | ✅（設計済み A-3） |
| 連続スライダー分布のシフト | ✅（A-4） |
| 両陣営コメントの構造・越境評価 | ✅ |
| 「中立層がどちらに動いたか」 | ✅ 集計 |

> **TwoSides Plus = 「この争点の世論と両論が、今どう動いてるか」が見える計器**

### Free vs Plus（案）

**Free**

- 争点記事、スプリット、ざっくり投票 %、トップコメント、投票・コメント

**Plus**

1. **層の動き** — 読了前後のシフト、「中立の N% が ○ 側に動いた」、時系列
2. **両陣営反応マップ** — 各側の論点クラスター、越境評価トップ、中立が引かれやすい論点
3. **レスバ支援 AI**（添え物）— 相手論理の弱点 + 争点素材からの反証候補（人格攻撃 NG）
4. **個人影響力** — 自分のコメントが N 人動かした、MVP 等

### レスバ支援 AI について

- FC の代わりに **「この争点で勝つための反論カード」**
- 争点に紐づいた素材のみ（Grok 一般検索との差）
- 論理・出典・事実のズレに限定（煽りツール化しない）

## 7. 他に FactBase から変えるべきもの

- **コピー**: 「一次情報にもとづいて」→「多ソースで両論を整理」（home, SITE, transparency）
- **カテゴリ**: ENTERTAINMENT / SOCIETY 等
- **モデレーション**: エンタメほど名誉毀損リスク高 → 強化
- **バッジラベル**: 「一次情報で確認」→「出典で確認」「報道・声明ベース」等
- **法令 DB ingest**: 主戦場ではなくなる（政治争点用に残す）

## 8. 技術メモ

- **Azure GPT-5**: 記事生成・AI 分析の軸。Gemini 不要（YouTube URL 直投入も不要）
- **YouTube Data API**: バズタイトルのみ。字幕 API はチャンネル OAuth が要るため使わない方針
- **既存資産**: スプリット UI、`Comment.stance`、`bridging.ts`、投票パネル、Radar パイプライン

## 9. 実装優先順位（更新）

### 供給（記事）

1. YouTube → 各局記事マッチ
2. スコープ緩和（声明対立型エンタメ）
3. `debatable` + promote 選定式
4. TwoSides 火種プロンプト + shareTitle
5. discover → promote 一本化の明文化

### 需要（議論・収益）

1. **投票 + 読前/読後** の計測（データ源泉）
2. **Plus ゲート**: 層の動き・両陣営マップ
3. 越境評価ランキング（スプリットの芯）
4. レスバ支援 AI（Plus 添え物）
5. FC oracle の降格・ラベル変更

## 10. 明示的にやらない

- 一次情報必須の記事公開
- YouTube 動画本文取得 / Gemini 依存
- 有料 FC を Grok 競合として売る
- 海外/食い違い/Wiki を記事化の必須条件にする
- 陣営の勝敗宣言・不安煽り
- 熱愛ゴシップ・試合結果のみの争点

## 11. 宣伝・GTM（2026-07-08 合意）

### プロダクトがやること（再確認）

**ゴシップ・論争の情報と意見を整理し、争点をわかりやすくする。**  
中立を保ちつつ、**議論に火をつける**（煽り・不安増幅ではない）。

### X 施策

| 案 | 内容 | 採用 |
|---|---|---|
| A. 全自動 Bot | 記事生成 → TwoSides カード → X 自動投稿 | 将来候補（品質リスク） |
| **B. 半自動（推奨）** | **記事は Radar で自動生成。X 投稿は良さそうなものだけ手動** | **当面これ** |

**フロー:**

```
Radar で争点記事を自動生成（discover → promote）
  → TwoSides カード（左右トップ論 + 投票 % + リンク）
  → 良さそうな争点だけ Founder が X に手動投稿
  → X のバズに乗っかってインプ・フォロワー
  → リンクでサイトへ → 投票・スプリット・議論
```

### ブランド上の鉄則（自動投稿する場合も共通）

- 人格を攻撃しない・特定アカウントに dunk しない
- **中立カードで水を配る**（炎上に gasoline を足さない）
- 検証通過 / HELD でない記事だけ外に出す
- 量より質。最初は **人間キルスイッチ必須**

### KPI

- X: インプ、フォロワー、リンク CTR
- サイト: PV → **投票率 → コメント率**（バニティ PV だけ追わない）
- 中長期: Plus（議論インテリジェンス）転換
