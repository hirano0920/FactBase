## 1. コンセプト — 第3のメディア

### 一行

**「偏向報道でも SNS のフィルターバブルでもない。中立な討論会場 — 第3のメディア。」**

### ユーザーが感じている痛み

| 場所 | 問題 |
|---|---|
| **オールドメディア** | 偏向報道、フレーム固定、読者は受け身 |
| **SNS（X・ヤフコメ・YouTube）** | フィルターバブル、同調圏の叫び、鬱になるアルゴ、出典なし |
| **従来まとめ** | 煽り・噂・匿名毒。議論の質がない |

### TwoSides が満たす欲求

人は消えない:

- **ゴシップもニュースも見たい**
- **議論したい**（「どう思う？」が見たい）
- **承認されたい**（自分の意見が通じた・役に立った）

TwoSides は **X やヤフコメを飛ばしてブックマークから来る** 習慣を作る。

```
初回: X のバズ → TwoSides で争点整理 → 投票 → スプリットを読む
2回目以降: ブックマーク / 通知 → 直接 TwoSides
```

**X は集客の通知装置。ホームは TwoSides。**

### 第3のメディアの定義

| | オールドメディア | SNS | **TwoSides（第3）** |
|---|---|---|---|
| 情報 | 編集部の一方向 | バズ・炎上・アルゴ | **多ソースで争点整理** |
| 意見 | コメント欄なし/弱い | 同じ側ばかり見える | **FOR / AGAINST 両方を意図的に見せる** |
| ランキング | なし | いいね＝同調 | **越境評価**（反対側からも「参考になった」） |
| アルゴ | 弱い | おすすめで心が沈む | **争点・新着・投票。鬱タイムラインなし** |
| 承認 | なし | 同調圏のいいね | **説得・越境・影響力** |

## 2. コンセプトを叶えるシステム

コンセプトはスローガンではなく、**以下のループと機能の組み合わせ**で実現する。

### 2.1 全体アーキテクチャ

```
┌─────────────────────────────────────────────────────────────┐
│  供給（AI + Radar・人件費ほぼゼロ）                          │
│  discover → promote → 多ソース収集 → 中立記事 → 公開         │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  体験（第3のメディアの芯）                                   │
│  争点記事 → 投票 → スプリット（FOR/AGAINST）→ 越境評価       │
│  読前/読後シフト → 沈黙の多数派分布 → 影響力/MVP            │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  拡散（広告費ゼロ）                                          │
│  TwoSides カード → X 手動投稿（当面）→ 流入 → ブックマーク化  │
│  SEO ロングテール（○○ 賛成 反対 / 声明 相違）                │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  収益                                                        │
│  無料: 広告 ｜ Plus/Pro: 議論インテリジェンス + 広告軽減/なし │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 供給 — Radar + AI（記事の自動製造）

| 要素 | 役割 | 実装 |
|---|---|---|
| **バズ検知** | 毎日「議論したい争点」を拾う | `discover.ts`（Trends / Yahoo / ニュース / YouTube タイトル） |
| **精査・公開** | 議論に値する話題だけ記事化 | `promote.ts`（buzzScore × debatable × 媒体数） |
| **多ソース収集** | 中立土台の素材 | `researchTopic()` + 各局 Web 記事（YouTube は検知のみ） |
| **記事生成** | 煽らず火種を作る | TwoSides 火種プロンプト（争点整理 + 両側論点） |
| **品質ゲート** | 薄い記事を外に出さない | 検証ループ + 人間キルスイッチ（初期） |

**記事の役割**: 真実を宣告する oracle ではなく、**両陣営が戦える共通の土台**。

**議論適性の必須条件**:

- 国内複数媒体（`distinctNewsOutlets >= 2`）
- 立場を取れる争点（`debatable`）
- 海外・Wiki・報道食い違いは **不要**

### 2.3 体験 — スプリット + 越境 + 計測

| コンセプト | システム | コード/設計 |
|---|---|---|
| 中立な記事 | 多ソース fire-starter、煽りなし | `radar-article.ts`, 火種プロンプト |
| 違う意見も見れる | スプリット FOR / AGAINST | `Comment.stance`, split UI |
| アルゴで煽らない | 争点一覧・新着・投票順 | おすすめタイムラインなし |
| 健全なランキング | **越境評価（bridging）** | `bridging.ts` — 反対側からの helpful が上 |
| 議論 | 投票 + コメント | vote-panel, comment-section |
| 承認 | 越境トップ・MVP・影響力 | A-1, A-3, ③（concept.md） |
| 沈黙の多数派 | 読了後スライダー分布 | A-4 ヒートマップ |
| コールドスタート | AI スティールマン（片側が空のとき） | A-2 |

**投票直後 UX**: 「あなたは賛成 → 反対側で最も参考になった意見はこちら」＝中立を体感させる瞬間。

### 2.4 拡散 — X 便乗 + SEO + ブックマーク

| チャネル | 役割 | 方針 |
|---|---|---|
| **X** | 新規争点の通知 | 記事は自動、**投稿は手動**（良い争点だけ）。dunk 垢は作らない |
| **TwoSides カード** | シェア単位 | 争点 + 投票% + 両陣トップ論 + リンク |
| **SEO** | 安定流入 | 「○○ 賛成 反対」「○○ 声明 相違」ロングテール |
| **ブックマーク** | リテンションの本体 | 争点フォロー・続報通知（C-5） |

### 2.5 インフラ — 低固定費

| 項目 | 選定 | コスト優位 |
|---|---|---|
| 配信・CDN | **Cloudflare** | 従量が安い、スケールしやすい |
| AI | **Azure AI Foundry** | スタートアップ支援・クレジット |
| DB / Redis | 既存構成 | Year 1 は月数万〜20万程度 |

**限界費用**: 記事は争点数 × API。PV 5000万でも AI 代は PV に比例しない。  
**粗利率**: Year 2 収益 6500万/月に対し、コスト 20〜80万/月 → **90%超**も可能。

## 3. 他メディアとの差別化

### 3.1 競合マトリクス

| 競合 | 強み | 弱点 | TwoSides の答え |
|---|---|---|---|
| **X / Grok** | リアルタイム、量 | フィルターバブル、出典なし、賛否不可視、鬱アルゴ | 争点整理 + 両側可視 + **投票・層の動き**（Grok に無い） |
| **ヤフコメ** | 反応が見たい | 毒、同調、検証なし | 越境評価 + スプリット |
| **YouTube コメ** | 動画連動 | 偏り、アルゴ、浅い | 記事土台 + 深い議論 |
| **Yahoo 要約** | 一目でわかる | 感情%、炎上増幅 | **理性分布**（読了後スライダー） |
| **まとめ（tweetsoku 等）** | PV・SEO・フック | 煽り、偏向、毒、有料理由なし | **参加型**（投票・スレ）+ **Plus 計器** |
| **オールドメディア** | 信頼・取材 | 偏向、議論の場がない | 多ソース中立 + **ユーザー議論** |

### 3.2 構造的に真似しにくい堀

1. **越境評価ランキング** — 「相手陣営も認めた意見が上」はヤフコメ/SNS と逆
2. **読前→読後の層の動き** — データが TwoSides にしかない
3. **スプリット + 投票 + 記事** が一体 — まとめは記事だけ、SNS は議論だけ
4. **Radar 自動供給** — 人力まとめの速度を AI で再現、中身は中立

### 3.3 普通のまとめで年8億が厳しい理由 → TwoSides なら筋が通る

| 普通のまとめ | TwoSides |
|---|---|
| 収益 = 広告 RPM のみ | 広告 + **Plus（議論計器）** |
| 次の煽りへ流す | 同じ争点で深掘り → 滞在・回遊 |
| 記者・編集の人件費 | Radar + AI → **限界費用ほぼゼロ** |
| 有料理由なし | 層の動き・両陣営マップ・影響力 |

## 4. マネタイズ — 2年目 年8億の現実的路線

### 4.1 目標の分解

**年8億 ≒ 月平均 6700万**

| 柱 | 月目標 | 年換算 |
|---|---|---|
| **広告** | **4000万** | 4.8億 |
| **サブスク（Plus + Pro）** | **2500万** | 3.0億 |
| **合計** | **6500万** | **7.8億** |

### 4.2 広告 4000万/月 — 逆算

```
広告収 = 月PV × RPM ÷ 1000
```

| RPM | 必要 月PV |
|---|---|
| ¥400（控えめ） | 1億 |
| ¥600 | 6700万 |
| **¥800（まとめ標準〜強め）** | **5000万** ← 中心シナリオ |
| ¥1000 | 4000万 |

**北極星: 月 PV 5000万 × RPM ¥800 ≈ 広告 4000万**

- 1日 約170万 PV
- まとめ並の広告枠（記事下・スレ間・ネイティブ）が前提
- 中立ブランドでも **参加型 UX で滞在が伸びれば RPM は確保可能**

### 4.3 サブスク 2500万/月 — 逆算

| プラン | 単価 | 必要人数（単独） |
|---|---|---|
| Plus | ¥500/月 | 5万人 |
| Pro | ¥980/月 | 2.6万人 |
| **混在（推奨）** | 実効 ¥625 前後 | **約4万人** |

**例**: Plus 3.5万人（1750万）+ Pro 7500人（735万）≈ **2485万/月**

**転換率の現実感**（MAU 300万想定）:

```
4万人 ÷ 300万 MAU = 1.3% 転換
```

5000万 PV / 月なら MAU 200〜500万は射程。**1% 前後の転換で 2500万/月**。

### 4.4 有料の芯 = 議論インテリジェンス（FC oracle ではない）

**Grok / X に無いデータ**:

| データ | TwoSides |
|---|---|
| 賛成 / 反対 / 中立の投票 | ✅ |
| 読前 → 読後の立場変化 | ✅ |
| 連続スライダー分布のシフト | ✅ |
| 両陣営論点クラスター・越境トップ | ✅ |
| 「中立層がどちらに動いたか」 | ✅ |
| 自分のコメントが N 人動かした | ✅ |

**FC は主役から降格** — ゴシップ争点では oracle として Grok に負ける。政治争点のみ RAG ボーナス。

#### Free vs Plus vs Pro

| | Free | Plus ¥500 | Pro ¥980 |
|---|---|---|---|
| 争点記事・投票・スプリット | ✅ | ✅ | ✅ |
| ざっくり投票 % | ✅ | ✅ | ✅ |
| **層の動き（読前/読後シフト）** | ❌ | ✅ | ✅ |
| **両陣営論点マップ・越境トップ** | 概要 | 詳細 | 詳細 |
| **個人影響力・MVP** | 簡易 | 詳細 | 詳細 |
| レスバ支援 AI | 味見 | ✅ | 多め |
| 広告 | あり | あり | **なし** |

**Plus = 「この争点の世論が今どう動いてるか」の計器**

### 4.5 2年ロードマップ（現実的フェーズ）

| | Year 1 前半 | Year 1 後半 | **Year 2** |
|---|---|---|---|
| **月 PV** | 50万 → 500万 | 500万 → 2000万 | **5000万** |
| **広告** | 50万 → 400万 | 400万 → 1600万 | **4000万** |
| **サブ** | 0 → 300万 | 300万 → 800万 | **2500万** |
| **月計** | ~100万 | ~2000万 | **~6500万** |
| **年計** | — | ~1.5億（通期） | **~8億** |

**Year 1 後半で月 2000万**（年2.4億ペース）まで行ければ、Year 2 の年8億は数字として破綻していない。

### 4.6 Year 2 までにやること（優先順）

#### 供給（記事を毎日回す）

1. YouTube → 各局 Web 記事マッチ
2. スコープ緩和（声明対立型エンタメ）
3. `debatable` + promote 選定式
4. TwoSides 火種プロンプト + shareTitle / voteQuestion
5. discover → promote 一本化

#### 体験（第3のメディアを体感させる）

1. スプリット + **越境評価ランキング**
2. **投票 + 読前/読後** 計測（Plus のデータ源泉）
3. TwoSides カード / OG（X シェア）
4. 沈黙の多数派ヒートマップ（差別化の看板）

#### 収益（2500万/月のサブ）

1. Plus ゲート: 層の動き・両陣営マップ
2. pricing コピーを「議論インテリジェンス」に刷新
3. 影響力・MVP 表示（承認欲求 → リテンション → 課金）
4. 3日間無料体験（Plus の計器を味見）

#### 集客（広告費ゼロ）

1. Radar 自動記事 + **X 手動投稿**（1日1〜3本）
2. SEO: 「○○ 賛成 反対」「○○ 声明」ロングテール
3. 争点フォロー通知 → **ブックマーク習慣**
4. 品質実証後に X 半自動化を検討

### 4.7 KPI（PV だけ追わない）

```
X インプ → リンク CTR → 争点 PV
  → 投票率
  → コメント率
  → ブックマーク / 直接流入比率
  → Plus 転換率
  → 広告 RPM
  → 月次 MRR
```

**成功の順序**: 参加率が上がる → MAU の質が上がる → Plus 転換 → PV スケール → 広告 4000万

## 5. リスクと対策

| リスク | 対策 |
|---|---|
| ブックマーク習慣が育たない | 毎日新争点（Radar）、通知、TwoSides カード |
| 中立が崩れると即死 | dunk しない、陣営勝敗宣言しない、左右比率監視 |
| AI 記事の品質 | ゴールデンセット評価、HELD ゲート、初期キルスイッチ |
| Google 薄コンテンツペナルティ | 質を固めてから量。スケールは増幅器 |
| Azure クレジット期限 | Year 1 中にモデル・争点数最適化 |
| サブ 2500万が遅れる | Plus ゲートを「層の動き」1点に集中、計器の体感を最優先 |
