# TwoSides — プロダクト構想・設計メモ

> 旧称 FactBase。スプリットスレッド（賛否を左右に並べる討論）を旗印にするため **TwoSides** へ改名予定。
> 本メモは 2026-07-07 の構想セッションの結論を固定したもの。実装の north star。

---

## 0. 一行コンセプト（2026-07-08更新: 第3のメディア）

**TwoSides = 第3のメディア。偏向もフィルターバブルもない、中立な討論会場。**

- 短: 「偏向報道でも、SNSのフィルターバブルでもない。第3のメディア — 中立な討論会場。」
- 少し長: 「TwoSidesは、オールドメディアの偏向とSNSのアルゴ鬱のあいだにある第3のメディア。中立な争点整理と投票、FOR/AGAINSTのスプリットで、自分と違う意見も見ながら議論できる。不要なおすすめはなく、議論と承認（越境評価）が得られる場所。」

### 何に対抗しているか

| | オールドメディア | SNS（X・ヤフコメ・YouTube） | TwoSides（第3） |
|---|---|---|---|
| 情報 | 偏向報道・フレーム固定 | バズ・炎上・アルゴ | 多ソースで争点整理 |
| 見え方 | 編集部の一方向 | 同じ意見ばかり（フィルターバブル） | FOR / AGAINST 両方 |
| 体験 | 読者は受け身 | 叫ぶ・煽る・鬱 | 投票・スプリット・議論 |
| 承認 | なし | いいね・リプ（同調圏内） | 越境評価（反対陣営からも「役に立った」） |
| アルゴ | なし/弱い | 不要なおすすめで心が沈む | 時系列・争点・投票。おすすめで煽らない |

**ターゲット**: オールドメディアの偏向とSNSのフィルターバブルの両方に挟まれてる人。Xやヤフコメは見たくないが、ゴシップもニュースも議論もしたい人がブックマークから来る。

### 人は議論と承認欲求から逃げられない

- **議論したい**（ゴシップもニュースも「どう思う？」が見たい）
- **承認欲求**（自分の意見が通じた・役に立った感）
- SNSは承認はあるが同じ側の人からだけ。メディアには議論の場がない。

TwoSidesなら:
- **スプリットスレッド** → 自分と違う意見も意図的に見える
- **投票** → 「世論の今」が数字で見える
- **越境評価（bridging）** → 反対側から「参考になった」が承認になる。同調圏のいいねより、越境の「役に立った」の方がこのプロダクトらしい承認

**なぜ「対決」でなく「解毒」のトーン**: 越境評価も沈黙の多数派も、本質は「相手を倒す」ではなく「対立そのものを冷ます」こと。声の大きさ・インプレゾンビが勝つのがX/SNSの病理で、TwoSidesは声のデカさでなく説得力が勝つ場所にする。反対意見を読むこと自体が、自分の意見を批判的に見つめ直すきっかけになる——これが「メディア不信×SNS疲れ」への解毒剤としての価値。デザイン・コピーのトーンも「対決・権威」でなく「ノイズからの解放・明晰さ」で作る。

### プロダクトとコンセプトの対応

| コンセプト | 機能 |
|---|---|
| 中立な記事 | Radar（多ソース・煽りなしfire-starter） |
| 違う意見も見れる | スプリットスレッド FOR/AGAINST |
| アルゴなし | 争点一覧・新着・投票結果。鬱タイムラインなし |
| 議論 | コメント・投票 |
| 承認 | bridgingランキング・影響力（Plus） |

年8億・X便乗・AI製造といったビジネスモデルは、この「第3のメディアであること」の上に載る。それがプロダクトの理由そのもの。

## 1. 戦略（なぜ勝てるか）

### 差別化の軸 = 「信頼」と「構造」（速度・量では戦わない）

| | 弱点 | TwoSidesの答え |
|---|---|---|
| X | 誤情報・扇動がアルゴリズムで増幅、出典なし、賛否が見えない | 一次情報グラウンディング＋賛否スプリット |
| ヤフコメ | 叩き合い・同調圧力、主張の真偽を誰も検証しない | 越境評価ランキング＋コメントFC |

### 需要 = メディア不信 × SNS疲れの受け皿
デマ対策以上に「**中立の地面**」への需要。アルゴリズムのフィルターバブル・炎上商法からの避難場所。中立層が「両側の一番賢い意見を並べて読める」体験は**日本では未出**。

### 注力順
1. コンテンツの信頼性と「自分ごと」度（入口の質＝最重要の堀）
2. FC精度（収益エンジン）
3. 両論が見えるUX（滞在価値）

### 最大の敵 = コールドスタート
議論は人が集まって初めて価値が出る。初期は「1人でも読んで賢くなれる／意見が整理できる」単体価値で集客し、議論は後から育てる。

## 2. 確定機能

### ① スプリットスレッド（左=反対 / 右=賛成）★命運を握る中核

- **データは既に揃っている**: `Comment.stance`（FOR/AGAINST/UNDECIDED）、`helpfulCount`。UI＋クエリで作れる（`prisma/schema.prisma` の Comment モデル参照）。
- 中立層が「両側の一番賢い意見を並べて読める」＝どこにもない体験。
- **投票直後に逆側の最強意見を提示**: 「賛成に投票 → 反対側で最も説得力ある意見はこちら👇」＝中立を体感させる瞬間。
- **片側が空なら「最初の一人になる👉」**: 過疎を投稿誘導に転換。
- 帰属意識・対決意識がエンゲージメントの燃料。

#### A-1. ランキングは「多数決」でなく「越境評価（ブリッジング）」★最重要
`helpfulCount` 単純順だと多数派が自陣を持ち上げて勝つ（＝ヤフコメ化）。そうではなく：

> **反対側の投票者からも「参考になった」を得たコメントを上に出す。**

Community Notes のブリッジング・アルゴリズムと同じ発想。「自分と逆の陣営すら認めた意見」こそ本物。`Comment.stance` と「helpfulを押した人のスタンス分布」の両方があるので実装可能。
**「相手陣営も唸らせた奴が勝つ」文化＝出典バトルをランキング指標に埋め込む。他サイトが構造的に真似しにくい最大の堀。**

→ 実装は最初からこのブリッジング設計で作る（後から入れ替えるより楽）。helpful に「押した人のスタンス」を記録する必要がある。

#### A-2. コールドスタートは「AIスティールマン」で埋める
意見が無い側を、AIが一次情報から**最も強い反対論**を生成して置く（人間の意見と明確に区別＝「AIによる論点提示」ラベル）。人間が書き込んだら自動で引っ込む/下がる。ブランド一致（AI×一次情報）＋中立読者は常に両論を見られる。

#### A-3. 「意見が変わった率」を可視化
記事を読む前の投票と、両論読了後の再投票を捕捉 →「この記事を読んで賛成→反対に変えた人 X%」。「中立の地面」の実証メトリクス。「最も意見を変えさせた争点」ランキングは新しい切り口。小規模な vote 履歴の追加が必要。

#### A-4. 沈黙の多数派ヒートマップ（読了後スペクトラム分布）★
SNS最大の苦痛＝「声のデカい1%を99%が見せられる」への回答。スプリットの中央に、**両論を読んだ後**の立ち位置を **-100〜+100 の連続スライダー**で取り、その**分布（山）を可視化**する。

```
強く反対 ─ やや反対 ─ 中立 ─ やや賛成 ─ 強く賛成
   ▁         ▃        █████████        ▃         ▁
  (1%)      (8%)         (82%)         (8%)      (1%)
```

真ん中が分厚い山になる＝「騒いでるのは両端の細い尻尾、冷静な多数派は真ん中」を一枚で証明。円グラフの「中立%」より視覚的に強い。

**設計の鉄則:**
- **3択の箱でなく連続スライダーの分布**で見せる（山の形が本体）。
- **「読了後」にだけ表示**して、"熟慮した中立"を"無関心/未読"と分離する（ここを混ぜると「99%どうでもいい」に見えて魔法が死ぬ）。
- **母数 n を必ず出す**。「TwoSidesでこの争点を見た人のうち」であって「国民の99%」ではない（過剰主張しない＝誠実さが武器）。
- Yahooの感情%と違い、**気分でなく理性的立場**を出す。じゃないと群衆増幅装置になる。
- A-3（読前→読後の移動）と合体すると「TwoSidesは人を過激化させず沈静化させる」データが取れる。

### ② 投稿前ファクトチェック ＋ 検証済みバッジ

- 既存 `factCheck(body, chunks)`（`src/lib/ai.ts`）は純粋関数。`retrieveChunks(issueId, draftBody)` → `factCheck(draftBody, chunks)` で**下書き本文のまま呼べる**（ほぼ移植）。
- 現状の `POST /api/comments/[id]/factcheck` は保存済みコメント向け。`commentId` はキャッシュキーに使うだけで判定には不要。
- **案B（採用）**: ブロックせず、矛盾を警告して自己修正を促す（検閲にしない）。
- **案C（採用）**: 通ったコメントに ✅検証済みバッジ → 課金者だけが検証機会を持つ → バッジがレスバの社会的証明 → 「勝つには✅が要る」で課金 → 質も上がる（**収益と品質が同方向**）。

#### バッジの鉄則（生命線）
- バッジは「チェックした」ではなく **「判定結果」に紐付ける**。
  - ✅ 一次情報で確認（TRUE） / 📄 条文上確認 のときだけ付く
  - 🔍 報道ベース（REPORTED）・⚖️ 議論あり（DISPUTED）は別表示
  - **FALSE はバッジなし**（＝修正を促す）／**OPINION は対象外**
- **課金で買えるのは「検証の機会」であって、バッジそのものではない。**
- **本文を編集したらバッジは即消える**（すり替え悪用防止）。判定は「その本文」に対するもの。
- 効果: レスバの土俵を「口の悪さ」から「**出典バトル**」へ変換＝健全な対決文化。

#### B-2. FCを「門番」でなく「調べ物アシスタント」に
裏取りできなかったとき、「この主張を支持/反証する一次情報はこれ」と RAG から提示。ユーザーは根拠を足してより良いコメントを書ける。FCが弾く道具でなく賢くする道具に → 課金価値が跳ねる。

#### B-3. 「信頼スコア」= 人気とは別軸のレピュテーション
既存レピュテーション（`src/lib/reputation.ts`）は likes 中心＝人気。そこに**「主張が検証を通った率」の実績**を別軸で足す（例:「このユーザーの主張は 87% が一次情報で裏取り済み」）。人気者でなく"正確な人"が評価される＝思想（真実 > 人気）の体現。継続課金の動機。

#### 課金ファネル
無料ユーザーがスレを見る → ✅付きの意見が正しく見える → 自分は付けられない（無料）→「勝つには✅が要る」→ 課金。無料層に一度だけ投稿前FCを味見させると転換率が上がる。

### ③ 「決着」の演出 — 勝者を決めず、変化を見せる ★

議論が煮詰まったときの"決着感"は強烈にバズる。だが **「陣営の勝敗」「AIが価値の勝者を宣言」は絶対にやらない**（中立の魂を殺す・負け組を疎外・偏向認定される。詳細はNon-goals）。代わりに、**同じドーパミンを"変化"と"個人の栄誉"で出す**。

原則: **「勝者を決めるな、変化を見せろ」。そして変化を"誰が起こしたか"まで見せる。**

#### 事実 vs 価値 の切り分け（前提）
- **事実の争い**（Xは本当か）→ 判定してOK（FCエンジンが TRUE/FALSE を出す＝勝敗でなく事実確認）。
- **価値の争い**（賛成すべきか）→ **勝者を宣言しない**。中立なデータを報告するだけ。

#### ③-1. MVPコメント（個人の栄誉）★絶対やる
- 議論が煮詰まったら **👑 この論題のMVP** をスポットライト表示。
- **なぜMVPかを数字で示す**: 「反対派だが、賛成派の54%も『参考になった』」＝越境評価（A-1）で格を証明。人気投票と一線を画す。
- プロフィールにトロフィー永続。争点横断の **MVP殿堂（Hall of Fame）** ＝回遊・SEO・シェア資産。

#### ③-2. 「あなたのコメントが N 人動かした」＝影響力の帰属 ★中核
立場変化（A-3）を**特定コメントに紐付ける**: 「このコメント投稿後、賛成が7%増えた」。
- **説得で人を動かした証拠**＝AIの審判なしの"決着"。「勝ち」と言わず「この主張が世論を動かした」と事実で示す。
- 「陣営が勝った」より「あなたが200人動かした」の方が**中毒性が高い**（宣言される勝利 < 勝ち取る影響力）。
- 個人レイヤーが本体: 負け組疎外なし・供給側（作り手）のリテンション・ユーザー自身がシェア（無料集客）。

#### ③-3. 変化の可視化演出（集団レイヤー＝文脈）
- before→after 分布（A-4ヒートマップ）が**ぬるっと変形**するアニメ。
- 「この議論が動かした人数」を主役の数字に。
- 賛成→反対等の**フロー図（Sankey）**でスクショ映え。

#### 新しい評価軸: 「影響力」（3軸目）
| 軸 | 意味 | 元 |
|---|---|---|
| 🎯 正確さ | 主張が一次情報で裏取りされた率 | FC（B-3） |
| ❤️ 人気 | いいね等 | 既存 |
| ⚡ **影響力** | **自分のコメントが動かした人数** | ★新規 |

罵倒でなく**説得が勝つ**という思想を、指標そのもので体現。MVP＝影響力軸の頂点。

## 3. 成長・拡散

### 🚀 TwoSidesカード（バイラルの配信単位）
どの争点も **1枚の共有画像**に自動変換：

```
［争点の問い］
賛成 52% ┃ 反対 48%
─────────╂─────────
賛成トップ論 ┃ 反対トップ論
（越境評価）  ┃（越境評価）
+ 出典
```

Xでシェアされる最小単位＝ブランド（左右＝TwoSides）を運ぶ。越境評価ランキング（A-1）とセットで、シェアするだけで中立性が伝わる。**成長エンジンとして最有力**。

### 🔁 自動フライホイール（記事自動生成 → X自動投稿）
```
Radar（話題検知）
  → 一次情報つき両論記事を自動生成（検証ループ通過）
  → TwoSidesカード＋引きのあるキャプション自動生成
  → Xに自動投稿（フック＋リンク）
  → 流入 → 投票・メーター参加 → FC課金
  → その盛り上がりが次のRadar検知に乗る
```
人力ゼロで回る集客エンジン。まとめが編集者を雇ってやることを、自動・出典付き・両論で回す。

**キャプションの引きの角度**（まとめ化せず面白く）: 両論の対立／沈黙の多数派（A-4）／感情 vs 事実／自分ごと。引きは強く、煽り・歪曲はしない。

**鉄の掟:**
- **検証を通った記事だけ自動投稿**（`generateVerifiedArticle` で HELD にならないもの）。薄い/未検証の自動投稿＝失敗をスケールさせる。
- **最初は人間のキルスイッチ付き**（遅延キュー/投稿前レビュー）。品質実証後に全自動。
- **自分の投稿はOK、他垢への自動リプはNG**（X自動化ルール＝凍結リスク）。
- 量より質のペース。キャプションにも名誉毀損ガード（報道ベース表現）。
- 自動投稿は品質の"増幅器"。**記事の質を固めた後の報酬**。

### 🎯 GTM: 「炎上した話題に中立カードで水を配る」
デマで両陣営が殴り合うトレンドに、中立の TwoSidesカードを出す（誰も攻撃しない＝敵を作らない）。特定アカウントに「突っ込む(dunk)討伐垢」はブランド（中立オアシス）と矛盾＋名誉毀損リスクなので**やらない**。人格は「淡々とした出典提供者」。もし個別の主張に触れるなら **人でなく主張に・出典付き・冷静に・1回で撤退**。左右は「一次情報と矛盾するか」の**単一基準**で扱い、自分の左右比率を監視。

### 🔎 SEO / まとめ流通戦略：「線路に乗る、まとめにならない」
まとめの数千万PVは**中身でなく流通の仕組み**で稼いでいる。流通は全部真似し、中身だけTwoSidesにする。
- **乗る（流通）**: SEOロングテール、Google Discover、SmartNews/Gunosy等アグリゲーター、アンテナ網、煽りでなく誠実な"引き"、24h人気＋新着の回遊、共有カード。
- **乗らない（中身）**: 噂・偏向・片側煽り・匿名毒。
- **SEOの狙い**:「みんなの反応」「○○ 反応」は巨大ボリュームなので狙う。ただし老舗まとめのドメイン権威が強いので、**序盤は「○○ 賛成 反対／論点／わかりやすく」など競合が手薄な高インテント長尾**で足場を作り、権威が育ってから「反応」に殴り込む。「反応」クエリでは**両側トップ論＋感情vs事実＋分布**という上位互換の答えを返して勝つ。「みんなの反応」はセクション名としても優秀（慣れた言葉で入口、中身で裏切る）。
- **武器**: Radarの自動量産（＝まとめ級の量とスピードを人力なしで）。**優位**: 出典・検証つき＝Google Helpful Content / YMYL で構造的に有利。
- **⚠️ 品質が先**: Google Helpful Content は薄い自動生成を激しく罰する。**スケールは質の増幅器**。薄いままスケール＝インデックス消失＝自殺。質を固めてから量に乗せる。
- **PVは目的でなく入口**: 怒りPVは広告RPMも低くFC課金に転換しない。KPIは「PV→参加→課金」の転換。バニティPVを追わない。

## 4. Backlog（差別化度が高い順・未確定）

- **C-1. 🏆「両陣営が認めた事実」ボックス**: 争点最上部に、賛成側・反対側の両方から高評価のコメント/事実を常設表示。A-1の越境評価をコンテンツ化。分断メディアの真逆＝看板機能候補。
- **合意の着地点をAIが提案**: 両論のコメントから「両者が飲めそうな中間案／争点の絞り込み」を生成。討論を分断で終わらせず統合に着地。C-1の能動版。
- **C-2. 📋 クレーム分解 × 各個FC**: 記事を3〜5個の争われる主張に分解し、各主張にFC判定＋出典。既存FC＋RAGの応用。
- **世論の時間変化**: 争点ごとの賛否を時系列表示。タイムライン（新一次情報の追加）と重ねて「新事実で世論が動いた」を可視化。vote＋`IssueTimeline` で低コスト。
- **C-4. 🧭 立ち位置診断（フィルターバブルの逆）**: 投票履歴から立ち位置を出し、「陣営内で割れている争点」「逆結論だが越境評価が高い意見」をレコメンド。あえて逆意見に触れさせる。
- **C-5. 🔔 争点フォロー＆続報通知**: `IssueTimeline`＋通知基盤があるので低コスト。retention 直結。
- **C-6. 🔗 全文が出典に飛べる読書体験**: 記事の各文を根拠チャンクにインラインリンク（claimsタグ＋チャンクは既存）。ラディカルな透明性。
- **ファクトクイズ・オンボーディング**: 「この主張、TRUE/FALSE？」→ 判定＋出典開示。教育的＆シェアされる集客surface。
- **争点間の相関マップ**: 「Aに賛成した人はBでこう投票」＝隠れた連合・意外な分断。データジャーナリズム的。
- **意見が変わった軌跡（個人版）**: 「あなたはこの3ヶ月で○回考えを変えた」＝オープンマインド度。意見を変えることを称賛＝反・部族主義。

## 5. 明示的に見送り（Non-goals）

- **公人トラッキング（発言と実績の照合）**: バズる可能性は極めて高いが、**「公約を破った」等の断定＝名誉毀損リスクが高すぎる**ため見送り。特定政党に偏って見えると即死する中立性リスクもある。将来やるとしても「本人の日付付き発言を並べるだけ（＝断定しない）」の無敵ゾーンに限定し、別ブランド＋データ基盤共有で慎重に。**当面は着手しない。**
- **陣営の勝敗判定 / AIによる価値の勝者宣言**: 価値の対立に客観的な正解はなく、AIが「こっちの勝ち」と言えば偏向認定で即死。負け組の疎外・ブリゲーディング誘発・沈黙の多数派の萎縮も招く。→ 代わりに §2-③「変化を見せる／個人の栄誉（MVP・影響力）」で決着感を出す。事実の争いのみ FC が判定してよい。
- **感情%の露出（Yahoo型）**: 「90%が怒り」を出すとバンドワゴンで炎上を増幅。TwoSidesの集約は"気分"でなく"読了後の理性的立場分布"（A-4）にする。
- **Xと速報の量／ヤフコメとコメント量で張り合う**こと。質と構造で差別化する。
- **デマ垢への討伐(dunk)アカウント**: 中立ブランドと矛盾＋名誉毀損・凍結リスク。集客は「中立カードで水を配る」形にする（§3 GTM）。

## 7. 記事クオリティの評価・改善 方法論

「本番アルゴリズムで何十個も作って改善」は正しいが、**目視だけはNG**（遅い・主観・リグレッションに気づけない）。**測れる評価ループ**で回す。

1. **ゴールデン・トピックセット（固定お題 20〜30）**: カテゴリ横断（法案/事件/国際/経済）、`OFFICIAL`/`REPORTED` 両方、速報型/スローバーン型。プロンプト変更前後を**同じお題でリンゴ対リンゴ比較**。
2. **評価ルブリック**（TwoSides専用）:
   - **両論性 ★最重要**（賛成・反対の最強論が公平に立つか＝記事の仕事はスプリットの土俵作り）
   - 事実正確性・出典整合（機械検証あり）／中立性・非煽り／自分ごと度／情報の厚み／構造・わかりやすさ／タイトルの引き
3. **3層評価**:
   - 自動（0円・既存）: `findUngroundedNumbers` `findUnclaimedHighlights` `violatesBan` 等
   - LLM-as-judge: 書き手と**別モデル**が各軸1〜5点＋理由（自己採点の循環回避）
   - 人間キャリブレーション: 創業者がサンプル採点しjudgeを較正（「面白い/自分ごと」の正解は人間の感覚）
4. **1変数ずつ反復**: ベースライン採点 → 最低軸を特定 → 1箇所だけ変更 → 同じセットで再生成 → スコア比較 → 良ければ採用。プロンプトはバージョン管理。
5. **オフライン（dry-run）で回す**: 本番DBを汚さず高速反復。評価スクリプト（例 `scripts/radar/eval-articles.ts`）にゴールデンセット生成→採点→スコアカードを固める。

**最初の一手**: 実在の多様な10〜15トピックで dry-run 生成 → 一緒に眺めて現状を可視化 → 実物に対しルブリック確定 → 評価ハーネス構築 → 反復。

## 8. 競合分析（学び / 差別化）

| 競合 | 何をしている | 弱点 | TwoSidesの差別化 / 学び |
|---|---|---|---|
| **X / Grok** | リアルタイム＋AI要約 | 単一投稿を事実誤認要約、出典なし、賛否不可視 | 一次情報＋複数ソース＋報道ベース表現＋検証ループ |
| **みんなの国会** | 法案×X投稿数ランキング（X API・7日集計） | 法案限定 | 学び: 話題ランキングの需要。※SmartNews運営ではない（`diet.ts` が使う `gian.json` が smartnews-smri のOSSというだけ） |
| **Yahooリアルタイム要約** | バズをAI要約＋感情%（ポジ/ネガ） | 出典なし・気分止まり・受け身・**炎上を増幅**・ゴシップ中心 | 学び: 一目の集約需要／「背景」セクション。差別化: 感情%でなく理性分布(A-4)、感情vs事実、参加型 |
| **まとめサイト**（例 tweetsoku） | 5ch/X反応を煽りタイトルで集約。日本最大級トラフィック | 噂・偏向チェリーピック・匿名毒・名誉毀損リスク・広告=炎上インセンティブ | **最重要の対抗軸**。学び: フックの型・回遊・代表コメント抽出・SEOロングテール。差別化: 出典/両側/参加/責任/"明確にするほど儲かる"収益構造 |

要点: これらは全て「**反応・意見が見たい**」という巨大需要の存在を証明している。今それを満たすのが「噂＋偏向＋煽り」なので、**中立・出典・両論の高台が空いている**。TwoSidesは「まとめの浄化版」。

## 9. 実装ロードマップ（Phase 0-7）

原則: ①薄い記事を増やさない ②供給 → 体験 → 計測 → 課金 → 拡散の順 ③既存資産（split/bridging/Radar 7割）を活かす

### Phase 0 — 供給パイプラインの穴埋め（最優先）✅ 実装完了(2026-07-09)
記事が出ないと全部意味がない。Radarは7割あるが、TwoSidesの争点（声明対立・YouTubeバズ）がフィルタで落ちている。

- **0-1 スコープ緩和（声明対立型エンタメ）** ✅: `filterRelevantTopics`のプロンプトに声明対立型の例外を明記し`debatable`フィールドを追加。`src/lib/buzz-prefilter.ts`のLIFESTYLE_HEADLINEから「不倫」「ハラスメント騒動」を外し、`src/lib/radar.ts`の`isOutOfScopeTopic()`に`DECLARATION_CONFLICT_OVERRIDE`（声明|反論|抗議|謝罪|否定|見解|コメントを発表|記者会見|訴訟|提訴|契約解除|解雇|懲戒）を追加。単なる熱愛・結婚・離婚報告（声明対立の無いもの）は引き続きreject
- **0-2 YouTube→各局Web記事マッチ** ✅: 新規`scripts/radar/lib/match-broadcast.ts`（`matchBroadcastArticle`/`mergeBroadcastMatch`）。YouTubeバズ由来のトピック（`buzz.inYouTubeTrending`）は主要放送局ドメイン(NHK/朝日/読売/毎日/産経/日経/各局ニュースサイト)限定でGoogle News検索し、見つかった記事を`evidence.news`に合流→既存の`fetchReportExcerpts`が本文取得。**未検証**: 実際のGoogle News RSSでの`site:`複数OR構文の挙動はライブ確認できていない（ローカルに外部API疎通環境なし）。本番cronで初回動作確認が必要
- **0-3 debatable判定+promote選定式** ✅: `SavedEvidence.debatable`追加。`weightedPromoteScore()`＝`buzzScore × (debatable===false ? 0.4 : 1.0) × min(1, distinctNewsOutlets/2) + twosidesFitBonus`を新設し、`selectTopicsForPromotion`のソートに適用（eligibility判定＝AND条件は変更せず、eligible内の並び順にのみ効く）。声明対立シグナル（`looksLikeDeclarationConflict`）で+2.5〜+3.0し、同程度〜やや低いbuzzでもTwoSidesお手本が1位寄りになる
- **0-3b debateType本線** ✅: `src/lib/debate-type.ts`。本線6型（declaration/policy/org_response/norm_flare/indicator/geopolitics）のみ promote。`filterRelevantTopics`が付与→discoverがevidence保存→promoteがゲート＋型別ボーナス→`generateArticle`が型別書き方ヒント。速報未確定・推定不能は出さない。policy+sustainedはreignite
- **0-4 discover→promote一本化の明文化** ✅: 確認の結果、detect.ts/discover.tsのヘッダーコメントは既に「バズ駆動はdetect.tsでpublishしない」「REPORTEDは国家的緊急のみ」を明記済みだった（想定と異なりコード変更は不要）。discover.tsのヘッダーとsrc/lib/radar-pipeline-inspect.tsの`IssueRoute`型に、rss_detect/summarizeは**恒久的な例外経路**（縮小・廃止予定ではない）である旨を追記して明文化のみ実施

完了条件: 声明対立型・YouTubeバズ話題が1日数本、安定的に記事化される（**本番cronでの実地確認は未実施**。0-1〜0-3はユニットテスト24件追加で単体動作は確認済み）
